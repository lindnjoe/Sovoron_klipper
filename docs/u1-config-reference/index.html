AFC_UnitType_1.cfg Configuration Overview

The AFC_UnitType_1.cfg file is used to define the configuration for the AFC system when using a specific unit type.

This file will typically be named in the format of AFC_Turtle_1.cfg for example, when a BoxTurtle is installed. If multiple unit types are installed, the file will be named in the format of AFC_Turtle_2.cfg, AFC_Turtle_3.cfg, etc.

Note

The actual naming convention may vary depending on your specific setup. The unit name can be renamed by using the install-afc.sh script.

[AFC_ACE unit_name] Section¶

The following options are available in the [AFC_ACE unit_name] section of the AFC_UnitType_1.cfg file. These options control the configuration of the AFC system when interfacing with Anycubic ACE PRO hardware. ACE units communicate over a serial connection and support two operational modes: combined (multiple slots share one toolhead path via a combiner) and direct (each slot feeds its own extruder independently).

Note

ACE and OpenAMS units are stepperless — they use their own internal firmware for filament movement instead of Klipper-controlled stepper motors. Lane sections for these units omit stepper pin configuration (step_pin, dir_pin, enable_pin).

Note

A virtual hub (switch_pin: virtual) is added to speed up tool change times by enabling two-phase loading — filament is pre-staged to the hub during prep, so only the hub-to-toolhead distance is traveled during a tool change. The dist_hub value defaults to 250 and will need to be manually set in your configuration to match your setup.

#    Optional: per-lane override for feed_length (mm).

#    Optional: per-lane override for retract_length (mm).

[AFC_OpenAMS unit_name] Section¶

The following options are available in the [AFC_OpenAMS unit_name] section of the AFC_UnitType_1.cfg file. These options control the configuration of the AFC system when interfacing with OpenAMS (Open Automatic Material System) hardware. OpenAMS units communicate with their own firmware and use an AFC_FPS buffer for filament pressure sensing.

OpenAMS inherits base unit configuration options (hub, extruder, buffer, LED colors, speeds) from the afcUnit base class, the same as BoxTurtle/NightOwl. Below are configuration values specific to OpenAMS.

Note

OpenAMS units do not support TurtleNeck-style (stepper-based) buffers. Only AFC_FPS buffers are compatible. If a non-FPS buffer is configured, it will be automatically disabled.

Tip

OpenAMS provides several calibration commands:

  • AFC_OAMS_CALIBRATE_PTFE UNIT=Oams_1 SPOOL=1 — Calibrate the PTFE tube length from the unit to the hub. Any lane can be used since the distance to the hub is identical for all bays. The result is saved to the [oams] config section.
  • AFC_OAMS_CALIBRATE_HUB_HES UNIT=Oams_1 SPOOL=1 — Calibrate the HUB Hall Effect Sensor value for a specific lane.
  • AFC_OAMS_CALIBRATE_HUB_HES_ALL UNIT=Oams_1 — Calibrate the HUB HES value for every loaded lane in the unit at once.

[AFC_FPS buffer_name] Section¶

The Filament Pressure Sensor (FPS) is a software-based buffer driver that uses an ADC (analog-to-digital converter) pin to read filament pressure. Unlike TurtleNeck buffers that use physical advance/trailing switches, FPS uses continuous analog readings to determine filament buffer state and adjust extruder rotation distance multipliers for smooth printing.

FPS buffers are required for OpenAMS units and can optionally be used with ACE units for pressure-based toolhead sensing. They can also be used with stepper-based units as an alternative to TurtleNeck buffers.

Note

FPS has PSF (Proportional Sync-Feedback) compatibility aliases: neutral_point maps to set_point, max_tension maps to low_point, and max_compression maps to high_point. You can use either naming convention.

ADC Configuration

[AFC_FPS FPS_1]
adc_pin: PA5
#    ADC pin for the FPS sensor. This parameter must be provided.
#    Use your MCU's ADC-capable pin (e.g. PA5, PC0, etc).
sample_count: 5
#    Default: 5
#    Number of ADC samples per measurement for noise averaging.
sample_time: 0.005
#    Default: 0.005
#    Time in seconds between ADC samples.
report_time: 0.100
#    Default: 0.100
#    ADC callback interval in seconds. Controls how often the FPS
#    value is updated.
reversed: False
#    Default: False
#    When True, reverses the ADC reading (1.0 - value). Use if your
#    sensor reads high when buffer is empty and low when compressed.

Buffer Tuning Parameters

# Buffer state thresholds
set_point: 0.5
#    Default: 0.5 (range: 0.1-0.9)
#    Target FPS reading — the ideal operating point. Buffer corrections
#    push the reading toward this value.
#    PSF alias: neutral_point
low_point: 0.1
#    Default: 0.1 (range: 0.0-0.5)
#    Threshold for buffer stretched / not feeding. Below this value,
#    the buffer is considered to be in tension (filament being pulled
#    faster than it's being supplied).
#    PSF alias: max_tension
high_point: 0.9
#    Default: 0.9 (range: 0.5-1.0)
#    Threshold for buffer compressed / filament loaded. Above this
#    value, the buffer is considered fully compressed.
#    PSF alias: max_compression

# Multiplier settings
multiplier_high: 1.15
#    Default: 1.15 (minval: 1.0)
#    Rotation distance multiplier when buffer is above set_point
#    (compressed state). Increases effective extrusion to relieve
#    buffer pressure.
multiplier_low: 0.85
#    Default: 0.85 (range: 0.0-1.0)
#    Rotation distance multiplier when buffer is below set_point
#    (stretched state). Decreases effective extrusion to let buffer
#    recover.
trailing_min_multiplier: 1.05
#    Default: 1.05 (minval: 1.0)
#    Minimum multiplier during trailing (compressed) state.

# Deadband and smoothing
deadband: 0.30
#    Default: 0.30 (range: 0.0-0.6)
#    Neutral window width centered on set_point. Within this band,
#    no correction is applied (multiplier stays at 1.0).
smoothing: 0.3
#    Default: 0.3 (range: 0.0-0.95)
#    Exponential moving average factor. Higher values give smoother
#    but slower response to changes.

# Timing
update_interval: 0.25
#    Default: 0.25 (minval: 0.05)
#    Correction timer interval in seconds. How often the multiplier
#    is recalculated based on the current FPS reading.
flip_cooldown: 180.0
#    Default: 180.0 (minval: 0.0)
#    Cooldown in seconds to suppress rapid direction changes between
#    advancing and trailing states.

# Error detection
filament_error_sensitivity: 0
#    Default: 0 (range: 0-10)
#    Error detection sensitivity. 0 = disabled. Higher values make
#    the buffer more aggressive at detecting filament errors.

# Misc
debug: False
#    Default: False
#    Enable debug logging for buffer state changes.
enable_sensors_in_gui: True
#    Default: (inherits from AFC global setting)
#    Show FPS buffer sensor in Mainsail/Fluidd GUI.

auto_spoolman_create¶

The auto_spoolman_create feature bridges RFID tag data with Spoolman inventory management. It works with ACE units (which have built-in RFID readers) and U1 lane scanners (which have RFID readers built into the spool holders). When a spool with an RFID tag is loaded or scanned, AFC reads the tag's embedded data — material type, color, brand, SKU, diameter, and temperature settings — then searches Spoolman for a matching filament and spool. If no match is found and auto_spoolman_create is enabled, AFC automatically creates the filament and spool entries in Spoolman so your inventory stays in sync without manual data entry.

How Matching Works

AFC uses a two-phase search to find existing Spoolman entries:

  1. SKU match — If the RFID tag contains a SKU (article number), AFC searches Spoolman for a filament with that exact SKU. An exact match is used immediately.
  2. Scored fallback — If no SKU match is found, AFC searches by brand and/or material and scores each candidate:
    • +3 points for matching material
    • +3 points for matching brand/vendor
    • +5 points for a close color match (color distance < 10), −6 for a poor match
    • +1 point for matching diameter (within 0.05mm)
    The highest-scoring candidate above a threshold of 4 is used.

When no suitable match exists and auto_spoolman_create: True, AFC creates a new filament entry (with vendor, material, density, diameter, color, and temperature settings from the tag) and a new spool entry linked to it. If the filament already exists but no spool is assigned, a new spool is created for that filament.

Configuration

auto_spoolman_create can be set at the unit level or overridden per-lane. The per-lane setting takes priority when present.

# Enable at the unit level (applies to all lanes in this unit)
[AFC_ACE ACE_1]
auto_spoolman_create: True

# Or override per-lane
[AFC_lane lane0]
unit: ACE_1:1
auto_spoolman_create: False
#    Disables auto-creation for this specific lane,
#    even though the unit has it enabled.

Note

This feature requires a working Spoolman instance connected through Moonraker. Without Spoolman configured, RFID tag data is still applied to the lane (material, color, temperatures) but no inventory entries are created.

spool_scanner¶

The spool_scanner feature turns a U1 lane into a dedicated RFID scanning station. Instead of loading filament, a scanner lane reads the RFID tag on a spool placed in it and stages that spool's data for the next filament load operation on any lane. This lets you scan a spool before loading it — the AFC system will know what material, color, and Spoolman spool ID to assign when the filament is actually loaded. This feature requires U1 RFID hardware.

How It Works

  1. A lane is configured with spool_scanner: true. AFC marks this lane as a scanner during initialization and begins actively polling its RFID reader.
  2. When a spool is placed on the scanner lane, AFC reads the RFID tag and extracts material, color, brand, SKU, and temperature data.
  3. If Spoolman is connected, AFC runs the standard matching logic (see auto_spoolman_create) and stages the matched or newly created spool ID as next_spool_id.
  4. If Spoolman is not connected, the raw tag data (material, color, temperatures) is stored as next_spool_info.
  5. When filament is next loaded into any lane, AFC automatically consumes the staged data — assigning the spool ID, material, color, and temperatures to that lane. The staged data is then cleared.

The scanner lane itself never loads filament. It exists solely to read tags and stage spool data for the next load.

Configuration

# Designate a lane as a spool scanner
[AFC_lane scanner_lane]
unit: Turtle_1:1
spool_scanner: true
map: T0
hub: Hub_1

SET_NEXT_SPOOL_ID Command

The SET_NEXT_SPOOL_ID G-code command provides manual control over the staged spool. This can be used in scanning macros, QR code workflows, or any automation that knows the Spoolman spool ID ahead of time.

# Stage a spool ID for the next load
SET_NEXT_SPOOL_ID SPOOL_ID=12345

# Clear the staged spool ID
SET_NEXT_SPOOL_ID

When a spool ID is staged via either the scanner lane or SET_NEXT_SPOOL_ID, it is consumed on the very next filament load and then cleared — subsequent loads are not affected unless a new spool is scanned or staged.

Tip

The spool scanner requires U1-compatible RFID hardware. If the scanned tag matches an existing spool in Spoolman, that spool is staged directly. If auto_spoolman_create is enabled on the scanner lane, new entries are created automatically when no match is found.

Example — FPS Buffer Setup¶

A fully-tuned FPS buffer configuration with LED indicators. FPS buffers use an ADC pin to continuously read filament pressure and adjust the extruder rotation distance multiplier in real time. This example shows all tunable parameters at their defaults.

[AFC_FPS FPS_1]
# --- ADC hardware ---
adc_pin: mcu:PA5
sample_count: 5
sample_time: 0.005
report_time: 0.100
reversed: False

# --- Buffer state thresholds ---
#   Filament pressure: 0.0 = no pressure, 1.0 = max compression
#
#   |  low_point   set_point   high_point  |
#   |     0.1    [---0.5---]     0.9       |
#   |  TENSION    NEUTRAL     COMPRESSION  |
#
set_point: 0.5
low_point: 0.1
high_point: 0.9

# --- Multiplier tuning ---
multiplier_high: 1.15
multiplier_low: 0.85
trailing_min_multiplier: 1.05
deadband: 0.30

# --- Response tuning ---
smoothing: 0.3
update_interval: 0.25
flip_cooldown: 180.0

# --- Error detection ---
filament_error_sensitivity: 0

# --- LED indicators ---
led_index: AFC_Indicator:1
led_buffer_advancing: 0,0,1,0
led_buffer_trailing: 0,1,0,0
led_buffer_neutral: 0,0.5,0.5,0
led_buffer_disable: 0,0,0,0.25

enable_sensors_in_gui: True
debug: False

Example — PSF Sensor Setup¶

The Proportional Sync-Feedback (PSF) sensor is an alternative naming convention for the AFC_FPS buffer. PSF uses neutral_point, max_tension, and max_compression in place of set_point, low_point, and high_point. Both naming conventions are fully supported and can be used interchangeably — the underlying buffer driver is the same.

# --- PSF sensor ---
[AFC_FPS my_buffer]
adc_pin: PA0
neutral_point: 0.5
#    Target pressure reading — the ideal operating point.
#    Equivalent to set_point.
max_tension: 0.1
#    Threshold for buffer under tension (filament being pulled
#    faster than supplied). Equivalent to low_point.
max_compression: 0.9
#    Threshold for buffer fully compressed. Equivalent to high_point.
